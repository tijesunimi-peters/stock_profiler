### Insider backfill — BLOCKED (code bug found, reporting for code track, not fixing)

Ran, completed in ~20s doing **nothing**:
```
2026-07-11 19:02:02,785 INFO insider backfill: 0 known issuer CIKs, limit=10
```
Exit code 0 — a **silent no-op**, no error surfaced.

**Root cause** (traced, not fixed — no source touched):

- `insider_backfill.py`'s `known_issuer_ciks()` (src/secfin/ingest/insider_backfill.py:60-67)
  restricts candidates to `RawFactRepository.get_ingested_ciks(BULK_SOURCE) |
  get_ingested_ciks(INCREMENTAL_SOURCE)`, which is `SELECT cik FROM ingest_checkpoint WHERE
  source = ?` (`src/secfin/storage/sqlite_repository.py:173-175`). This is a deliberate safety
  filter per the module's own docstring (avoid mis-attributing reporting-owner-only filer CIKs
  as issuers) — reasonable in principle.
- On THIS live DB, `ingest_checkpoint` has **0 rows** (verified directly: `select count(*) from
  ingest_checkpoint` → 0), while `raw_facts` has **395,527 rows across 6,736 distinct CIKs**
  (verified: `select count(distinct cik) from raw_facts` → 6736).
- Why the mismatch: `ingest.backfill`/`ingest.incremental` are the only writers of
  `ingest_checkpoint`, and they do so atomically with their fact writes via
  `upsert_raw_facts_and_checkpoint` (`sqlite_repository.py:134-147`, one transaction, by
  design, per its own docstring: "a crash can't leave one without the other"). But the live
  API's cache-aside path (`api/routes.py`'s `_facts_for_cik` / `_statement_facts_for_cik`) calls
  the OTHER method, `repo.upsert_raw_facts(facts)` (routes.py:201, 233) — facts only, **no
  checkpoint row**. Every company on this DB got its `raw_facts` rows through live on-demand
  reads (other tracks' spot-checks, e2e/load-test runs, dev usage), never through an actual
  `python -m secfin.ingest.backfill` or `.incremental` run — so `ingest_checkpoint` legitimately
  has nothing in it even though the company universe is fully populated.
- Net effect: `insider_backfill`'s candidate-discovery assumption ("checkpoint rows ==
  known-good issuers") silently degrades to the empty set on any DB that reached its current
  state mainly via live cache-aside traffic rather than a completed bulk backfill. This looks
  like exactly the deployment history of this DB.

**Why I didn't work around it:** the only workarounds available to me without touching source
(e.g., hand-inserting rows into `ingest_checkpoint` with `source='bulk_companyfacts'` or
`'daily_incremental'`) would falsely mark ~6,736 CIKs as "already bulk-ingested," which would
then make a FUTURE real `ingest.backfill`/`.incremental` run skip them — corrupting the exact
resumability mechanism CLAUDE.md calls out as load-bearing. That's a data-integrity risk I'm not
willing to take from the data lane; it's a call for the code track (fix
`known_issuer_ciks()` to also consider `all_ciks()`, or decide the real fix is running
`ingest.backfill` properly first).

  after successfully processing thousands of candidates:
  ```
  File ".../secfin/ingest/institutional_backfill.py", line 135, in _process_candidate
      repo.upsert_snapshot(snapshot)
  File ".../secfin/storage/sqlite_holdings_repository.py", line 157, in upsert_snapshot
      self._conn.executemany(...)
  sqlite3.IntegrityError: UNIQUE constraint failed: holdings_other_managers.manager_cik,
  holdings_other_managers.report_period, holdings_other_managers.sequence_number
  ```
  Last candidate in the log before the crash: **manager CIK 1890906, accession
  0001890906-26-000040** (filing document `13F_FinalQ12026.xml`). I verified the transaction
  rolled back cleanly — `select * from holdings_snapshots where manager_cik=1890906` returns
  **no row** — so the DB is not corrupted, just missing that manager and everything after it in
  scan order.

  **Root cause, read-only (no source touched):** `_process_candidate`
  (`src/secfin/ingest/institutional_backfill.py:114-136`) only wraps the *fetch* call
  (`fetch_13f_snapshot_for_filing`, lines 127-133) in `try/except Exception`; the subsequent
  `repo.upsert_snapshot(snapshot)` on line 135 is **outside** that block. `upsert_snapshot`
  (`src/secfin/storage/sqlite_holdings_repository.py:110-157`) does correctly wrap its own SQL
  in `BEGIN`/`COMMIT`/`ROLLBACK` (so a bad snapshot can't corrupt the DB), but it re-raises after
  rollback, and nothing above it catches that re-raise. So a single manager's malformed
  `other_managers` roster (evidently CIK 1890906's cover page lists two co-filing managers under
  the same `sequenceNumber` — a real-world EDGAR data quirk the parser doesn't dedupe) takes
  down the entire batch job, not just that one candidate.
  **This means the job cannot be completed by simply rerunning it**: candidate order is
  deterministic (zip `namelist()` order), CIK 1890906's data hasn't changed, and it was never
  cached (rolled back) — so a rerun would reprocess everything already-cached as instant skips,
  then die at the *exact same candidate* again, never reaching the ~2,040 candidates
  (8,803 total − 6,759 attempted before the crash) that were never even attempted this run.

- Not reattempting `institutional_backfill` myself — a rerun would burn real throttled SEC
  requests re-fetching ~6,736 already-good managers only to crash at the same spot again,
  making zero net progress on the untouched tail. Flagging both issues (the 22 validation
  failures and, more importantly, the unhandled-IntegrityError crash) for the **code track**:
  the fix is presumably (a) widen `_process_candidate`'s `try/except` to also cover
  `repo.upsert_snapshot`, tallying that outcome as `"failed"` like the fetch-side errors, so one
  bad manager doesn't kill the run, and separately (b) dedupe/tolerate duplicate
  `sequenceNumber`s when parsing `otherManagers2Info` (or drop the `sequence_number` uniqueness
  constraint's strictness) so CIK 1890906 itself can be stored. Per my lane, not fixing either
  myself.
- **Net result for launch readiness:** Q1 2026 (`2026-03-31`) 13F coverage went from 5 managers
